Задачи для тренировки Java (начальный уровень)
Калькулятор с юнит‑тестами
Напишите класс Calculator с методами: add(a, b), subtract(a, b), multiply(a, b), divide(a, b).
Создайте тесты для каждого метода (используйте JUnit 5).
Проверьте: деление на ноль должно выбрасывать ArithmeticException.

Работа с массивами
Напишите метод, который находит:
максимальное значение в массиве;
среднее арифметическое;
количество чётных чисел.
Добавьте тесты для крайних случаев (пустой массив, отрицательные числа).

Строковые операции
Реализуйте метод, который:
переворачивает строку;
проверяет, является ли строка палиндромом;
считает количество гласных букв.
Напишите тесты для Unicode‑символов (например, «ё», «ü»).

Класс «Студент»
Создайте класс Student с полями: name, age, grades (список оценок).
Добавьте методы:
getAverageGrade() — среднее по оценкам;
isHonorStudent() — возвращает true, если средний балл ≥ 4.5.
Напишите тесты для граничных значений (например, пустой список оценок).

Обработка исключений
Напишите метод, который принимает строку и пытается преобразовать её в число.
Если строка не числовая — выбросьте NumberFormatException с пояснением.
Напишите тест, проверяющий обработку исключения.
Циклы и условия
Реализуйте метод, который выводит на экран:
все числа от 1 до 100, кратные 3;
«Fizz» вместо чисел, кратных 5;
«Buzz» вместо чисел, кратных 7;
«FizzBuzz» для чисел, кратных и 5, и 7.
Напишите тесты для проверки вывода.

Работа с коллекциями
Создайте список строк. Напишите методы:
удалить все строки короче 5 символов;
отсортировать по длине;
найти самую длинную строку.
Добавьте тесты для пустого списка и списка с одинаковыми строками.

Мини‑игра «Угадай число»
Программа загадывает число от 1 до 100.
Пользователь вводит предположение.
Программа отвечает: «Больше», «Меньше» или «Угадали!».
Добавьте тест, который имитирует ввод пользователя через Scanner (используйте моки, например, Mockito).

Дата и время
Напишите метод, который:
принимает дату в формате yyyy-MM-dd;
возвращает день недели (например, «Понедельник»);
проверяет, является ли дата выходным.
Напишите тесты для високосных годов и крайних дат.

Чтение/запись в файл
Создайте метод, который:
записывает список строк в файл;
читает файл и возвращает список строк;
обрабатывает FileNotFoundException.
Напишите тесты с временными файлами (используйте Files.createTempFile()).

Советы по тренировке
Пишите тесты до кода (подход TDD).

Используйте JUnit 5 (актуальная версия).

Для сложных задач применяйте Mockito (моки) и AssertJ (удобные assert‑проверки).

Проверяйте покрытие кода тестами (в IDEA: Run → Run with Coverage).

Старайтесь, чтобы каждый тест проверял ровно одну вещь.

## Подход TDD на практике: пошаговое руководство


TDD (Test‑Driven Development) — это **цикл из трёх повторяющихся шагов**: «красный → зелёный → рефакторинг».


### Цикл TDD

1. **Красный (Red)**  
   - Напишите тест для новой функциональности.  
   - Тест **обязательно должен провалиться** (ведь кода ещё нет!).  
   - Если тест прошёл — либо функциональность уже есть, либо тест некорректен.

2. **Зелёный (Green)**  
   - Напишите **минимальный код**, чтобы тест прошёл.  
   - Не добавляйте лишнего — только то, что требуется для прохождения теста.  
   - Цель: сделать тест «зелёным» любой ценой (даже если код «корявый»).


3. **Рефакторинг (Refactor)**  
   - Улучшите код: сделайте его чище, понятнее, эффективнее.  
   - **Важно:** тесты должны продолжать проходить!  
   - Удалите дублирование, улучшите имена переменных, оптимизируйте алгоритмы.


### Практический пример: калькулятор сложения


**Шаг 1. Красный: пишем тест (JUnit 5)**
```java
import static org.junit.jupiter.api.Assertions.assertEquals;
import org.junit.jupiter.api.Test;

class CalculatorTest {
    @Test
    void shouldAddTwoNumbers() {
        Calculator calculator = new Calculator();
        int result = calculator.add(2, 3);
        assertEquals(5, result);
    }
}
```
→ Тест падает (класса `Calculator` ещё нет).


**Шаг 2. Зелёный: пишем минимальный код**
```java
class Calculator {
    int add(int a, int b) {
        return a + b; // Просто возвращаем сумму
    }
}
```
→ Тест проходит!

**Шаг 3. Рефакторинг**  
- Код уже прост и понятен — рефакторинг не требуется.  
- Если бы метод был сложнее, можно было бы:  
  - добавить валидацию входных данных;  
  - разбить на подметоды;  
  - улучшить читаемость.


### Ключевые принципы TDD

1. **Тесты пишутся ДО кода**  
   - Это заставляет чётко сформулировать требования.  
   - Вы сразу видите, что должно работать.

2. **Минимальный код для прохождения теста**  
   - Не пытайтесь предугадать будущее — реализуйте только то, что нужно сейчас.  
   - Избегайте «умного» кода на этапе «зелёного».

3. **Частые итерации**  
   - Цикл должен быть быстрым (минуты, а не часы).  
   - Чем меньше шаг — тем проще отлаживать.


4. **Тесты — это документация**  
   - По тестам можно понять, как работает код.  
   - Названия тестов должны быть понятными (например, `shouldAddTwoNumbers`).

5. **Рефакторинг без страха**  
   - Тесты гарантируют, что вы не сломаете существующую функциональность.  
   - Можно смело улучшать архитектуру.


### Инструменты для TDD в Java

- **JUnit 5** — основной фреймворк для юнит‑тестов.  
- **Mockito** — для создания моков (имитации зависимостей).  
- **AssertJ** — удобные assert-методы с читаемыми сообщениями.  
- **Coverage‑плагины** (в IDEA) — показывают, какие строки кода покрыты тестами.


### Типичные ошибки новичков

1. **Писать код до тестов**  
   → Нарушает суть TDD.  
2. **Пытаться покрыть всё сразу**  
   → Начинайте с малого: один тест = одна маленькая функция.  
3. **Игнорировать рефакторинг**  
   → Код быстро становится «грязным».  
4. **Писать слишком сложные тесты**  
   → Тест должен проверять одну вещь.  
5. **Не запускать тесты регулярно**  
   → Лучше запускать их после каждого изменения.


### Когда TDD особенно полезен

- Бизнес‑логика (валидация, расчёты).  
- Утилитные классы (работа с датами, строками).  
- API‑сервисы (контроллеры, сервисы).  

### Когда стоит подумать дважды


- Работа с внешними API (требует моков).  
- GUI‑приложения (сложно тестировать интерфейс).  
- Нагрузочные тесты (TDD не заменяет performance‑тестирование).


### Советы по внедрению

1. Начните с простого проекта (например, калькулятор).  
2. Используйте IDE с поддержкой TDD (IntelliJ IDEA, Eclipse).  
3. Пишите тесты в том же пакете, что и код (обычно `src/test/java`).  
4. Дайте тестам понятные названия (например, `shouldThrowExceptionWhenNullInput`).  
5. Запускайте тесты перед коммитом.


**Итог:**  
TDD — это не магия, а дисциплина. Он требует привычки, но окупается:  
- меньше багов;  
- проще рефакторинг;  
- понятная документация через тесты;  
- уверенность в коде.
